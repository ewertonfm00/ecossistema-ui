## Story

**Como** desenvolvedor de um módulo consumidor do ecossistema,
**quero** ver composições realistas de telas completas no Storybook do `@aiox/ui`,
**para que** eu entenda como compor AppShell + SidebarNav + PageHeader + componentes de conteúdo em cenários reais sem precisar experimentar por tentativa e erro no meu módulo.

## Contexto

Stories 0.2–0.5 entregaram átomos, moléculas, organismos e documentação individual de cada componente. O que falta são **composições de página** — telas completas que mostram como os componentes interagem no contexto real de uma aplicação.

Esta story cria 5 "page stories" em `src/pages/` que demonstram composições típicas do ecossistema (login, dashboard, listagem, formulário, empty state). São apenas para Storybook — não exportadas no `src/index.ts`.

**Regra fundamental:** todos os dados são mockados. Sem estado externo, sem fetch, sem React Router. As páginas são componentes React puros.

### 2. Parâmetros globais das stories

Todas as stories de página usam:

```tsx
parameters: {
  layout: 'fullscreen',
}
```

Isso é obrigatório porque `AppShell` usa `h-screen` e precisa do viewport completo para renderizar corretamente.

### 3. `LoginPage` — formulário de autenticação

**Composição:** AppShell sem sidebar (layout centrado) + Card + FormField + Input + Button + Label

**Props:**
```ts
interface LoginPageProps {
  onSubmit?: (data: { email: string; password: string }) => void
  isLoading?: boolean
  error?: string
}
```

**Layout:**
- AppShell sem sidebar — usar `sidebar={<></>}` ou variante centrada:
  ```tsx
  // LoginPage não usa o slot sidebar — layout centralizado no main
  <div className="flex h-screen items-center justify-center bg-neutral-50">
    <Card className="w-full max-w-md p-8">
      ...
    </Card>
  </div>
  ```
  **Nota:** LoginPage é o único caso que não usa AppShell + SidebarNav — é uma tela standalone centrada.

**Conteúdo do Card:**
- Logo/título: `<h1>` com "Entrar" ou "Clínica Premium" (texto mockado)
- `<FormField label="E-mail">` com `<Input type="email" placeholder="seu@email.com" />`
- `<FormField label="Senha">` com `<Input type="password" placeholder="••••••••" />`
- `<Button variant="primary" className="w-full">Entrar</Button>`
- Link mockado "Esqueci minha senha" como `<a>` com `text-primary-600`
- Se `error` fornecido: `<Alert variant="error">{error}</Alert>` acima dos campos

**Story variants:**
- `Default` — formulário em branco
- `Loading` — `isLoading={true}` (Button disabled/loading state)
- `WithError` — `error="Credenciais inválidas. Tente novamente."`

**Testes (mínimo 2):**

## Notas técnicas

### `parameters: { layout: 'fullscreen' }` é obrigatório

`AppShell` usa `h-screen` e `overflow-hidden`. Sem `fullscreen`, o Storybook adiciona padding ao redor e o layout quebra.

### LoginPage sem AppShell

LoginPage é a única exceção — é uma tela de autenticação sem sidebar. Layout: `div.flex.h-screen.items-center.justify-center` envolvendo um `Card` centralizado.

### Dados mockados — organizados em arquivo separado

Mocks pesados (como os 15 clientes da ListingPage) podem ser extraídos para `__mocks__/` dentro de cada pasta de página ou inline no arquivo `.stories.tsx`. A escolha é do `@dev` — qualquer abordagem é válida.

### Testes — escopo mínimo intencional

Os testes de páginas são smoke tests — apenas garantem que a composição não quebra. Testes de interação dos componentes individuais já existem nas stories 0.2–0.4.

## Valor de negócio

Page compositions são a principal referência para devs dos módulos. Em vez de adivinhar como compor AppShell + SidebarNav + DataTable + PageHeader, o dev consulta o Storybook e vê a tela completa funcionando. Reduz tempo de onboarding e elimina inconsistências de layout entre módulos.

| Risco | Probabilidade | Mitigação |
|-------|--------------|-----------|
| `AppShell` com `h-screen` causar overflow no Storybook sem `fullscreen` | Alto | `parameters: { layout: 'fullscreen' }` obrigatório em todas as stories |
| SidebarNav items usando icons que não existem no `Icon` átomo | Médio | Verificar lista de icons suportados no `Icon.tsx` antes de usar; substituir por icons disponíveis |
