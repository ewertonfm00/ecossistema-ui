## Story

**Como** desenvolvedor de um módulo consumidor do ecossistema,
**quero** que o `AppShell`, `SidebarNav`, `PageHeader` e `DataTable` sejam responsivos com breakpoints Tailwind bem definidos,
**para que** as telas construídas com `@aiox/ui` funcionem corretamente em dispositivos mobile (≥ 375px) sem que os módulos precisem reescrever o layout.

---

## Contexto

Todos os organismos do Épico 03/04 foram implementados pensando em desktop. O `AppShell` renderiza a sidebar lateral sempre visível, o `DataTable` não tem scroll horizontal e o `PageHeader` não adapta breadcrumbs em telas pequenas.

O público-alvo (gestoras de clínicas de estética) acessa predominantemente via mobile. O ecossistema precisa de responsividade nativa no design system — os módulos não devem ter que corrigir layout em cada tela.

**Decisão arquitetural:**
- Breakpoint de colapso da sidebar: `lg` (1024px) — abaixo disso, sidebar vira `Drawer`
- O `Drawer` do Épico 04 (Story 2.0) é reutilizado como sidebar mobile — zero reimplementação
- `AppShell` recebe um `useSidebar` hook interno (exportado) para que os módulos controlem abertura/fechamento
- Sem mudança de API pública dos componentes — apenas novos props opcionais; zero breaking change

### 2. Hook `useSidebar`

### 3. `AppShell` — sidebar responsiva

#### 3a. Novos props (todos opcionais — zero breaking change)

```ts
interface AppShellProps {
  // props existentes mantidas sem alteração
  sidebar: React.ReactNode
  header?: React.ReactNode
  children: React.ReactNode
  // NOVOS props:
  sidebarOpen?: boolean          // controle externo da sidebar mobile; padrão: false
  onSidebarClose?: () => void    // callback ao fechar o drawer mobile
  sidebarWidth?: string          // largura da sidebar desktop; padrão: 'w-64'
}
```

#### 3b. Comportamento desktop (≥ lg / 1024px)

- Sidebar permanece visível no lado esquerdo (comportamento atual — sem mudança)
- `sidebarOpen` e `onSidebarClose` são ignorados em desktop
- Layout: `flex flex-row` com sidebar fixada à esquerda

#### 3c. Comportamento mobile (< lg)

- Sidebar desktop fica oculta (`hidden lg:flex`)
- Um `Drawer` (Story 2.0, `placement="left"`) é renderizado com o mesmo conteúdo `sidebar`
- `Drawer` é controlado por `sidebarOpen` / `onSidebarClose`
- O `header` (slot) deve ter espaço reservado para um botão de menu (hamburger) — o `AppShell` não renderiza o botão diretamente; cabe ao módulo inserir no slot `header`
- Overlay do Drawer cobre o conteúdo principal

#### 3d. Estrutura JSX resumida

```tsx
<div className="flex flex-col min-h-screen">
  {header && <div className="...">{header}</div>}
  <div className="flex flex-1 overflow-hidden">
    {/* Desktop sidebar */}
    <aside className={cn('hidden lg:flex flex-col', sidebarWidth, 'border-r border-neutral-200 bg-white')}>
      {sidebar}
    </aside>

    {/* Mobile sidebar — Drawer */}
    <Drawer
      open={sidebarOpen ?? false}
      onClose={onSidebarClose ?? (() => {})}
      placement="left"
      size="sm"
      title=""
    >
      {sidebar}
    </Drawer>

    {/* Conteúdo principal */}
    <main className="flex-1 overflow-auto p-4 lg:p-6">
      {children}
    </main>
  </div>
</div>
```

### 5. `PageHeader` — breadcrumb colapsível em mobile

**Arquivo:** `src/organisms/PageHeader/PageHeader.tsx` (existente)

#### 5a. Comportamento

- Em mobile (< md): exibe apenas o último item do breadcrumb com um `…` antes (ellipsis)
  - Exemplo: `Home > Clientes > Novo` → `… > Novo`
- Em desktop (≥ md): breadcrumb completo (comportamento atual)
- A lógica de colapso é CSS-only (sem JS): os itens do meio têm `hidden md:inline`

#### 5b. Implementação

```tsx
{breadcrumbs.map((crumb, i) => (
  <span key={i}>
    {i > 0 && i < breadcrumbs.length - 1 ? (
      // itens do meio: ocultos em mobile
      <span className="hidden md:inline">
        <a href={crumb.href}>{crumb.label}</a>
        <span className="mx-1 text-neutral-400">/</span>
      </span>
    ) : i === 0 && breadcrumbs.length > 2 ? (
      // primeiro item: em mobile vira '…'
      <span>
        <span className="md:hidden text-neutral-400">…</span>
        <span className="hidden md:inline"><a href={crumb.href}>{crumb.label}</a></span>
        <span className="mx-1 text-neutral-400">/</span>
      </span>
    ) : (
      // último item: sempre visível
      <span>{crumb.label}</span>
    )}
  </span>
))}
```

**Stories — adicionar:**
- `Mobile` — viewport 375px com breadcrumb de 4 níveis

**Testes (mínimo 1 novo):**
- [ ] Com 3+ breadcrumbs, os itens do meio têm classe `hidden md:inline`

## Escopo negativo

- ❌ Dark mode — fora de escopo global
- ❌ `useSidebar` com persistência no `localStorage` — estado apenas em memória
- ❌ `DataTable` com card view em mobile (troca de layout) — apenas scroll horizontal
- ❌ `AppShell` com sidebar redimensionável via drag — fora de escopo
- ❌ `SidebarNav` com submenu em modo compact — apenas itens de nível 1
- ❌ `PageHeader` com breadcrumb colapsível via clique em `…` — apenas ellipsis visual
- ❌ Testes de responsividade com Playwright ou cypress — escopo é Vitest + JSDOM
- ❌ Animações com `framer-motion` — CSS/Tailwind apenas
- ❌ `AppShell` com header sticky — slot `header` é responsabilidade do módulo

## Notas técnicas

### AppShell — não reimplementar o Drawer

O `Drawer` da Story 2.0 já implementa overlay, trap de foco, `role="dialog"`, `aria-modal`, fechamento por Escape e portal. O `AppShell` importa e reutiliza — zero duplicação de lógica.

### useSidebar — hook simples, sem Context

O `useSidebar` é um hook local que retorna `{ isOpen, open, close, toggle }`. Não usa Context porque cada `AppShell` gerencia seu próprio estado — se o módulo precisar de estado global, ele mesmo cria o contexto.

### DataTable — sticky header requer container com height

`position: sticky` no `<thead>` só funciona se o container pai tiver `overflow-y: auto` com altura definida. Adicionar `max-h-[600px] overflow-y-auto` ao `div.overflow-x-auto` para que o sticky funcione quando a tabela for alta.

### PageHeader — ellipsis sem JS

A solução CSS-only (`hidden md:inline`) é preferível a `window.innerWidth` em JavaScript porque:
1. Funciona em SSR sem hidratação
2. Não quebra em resize sem listener
3. Tailwind já remove classes não usadas no bundle

### SidebarNav compact — não remover labels do DOM

Em modo `compact`, os labels devem estar no DOM mas visualmente ocultos (`sr-only`) para que screen readers ainda anunciem o item. Não usar `aria-hidden="true"` nos labels.
