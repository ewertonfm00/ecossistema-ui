## Story

**Como** desenvolvedor implementando qualquer módulo do ecossistema (Auth, Estética IA, Marketing, Gestão, ML Lab),
**quero** moléculas prontas no `@aiox/ui` para compor formulários, exibir cards de conteúdo e mostrar feedback ao usuário,
**para que** eu possa construir telas funcionais com consistência visual sem recriar padrões compostos do zero em cada módulo.

## Contexto

Story 0.2 entregou os 7 **átomos** (Button, Input, Label, Badge, Avatar, Icon, Spinner). Esta story entrega as 4 **moléculas core** — componentes compostos que combinam esses átomos em padrões reutilizáveis que aparecem em 100% das telas do ecossistema:

- **FormField** — padrão universal de campo de formulário
- **Card** — container de conteúdo
- **Alert** — feedback inline persistente
- **Toast** — notificação temporária com auto-dismiss

**Regra de composição:** moléculas usam APENAS os 7 átomos da Story 0.2 e elementos HTML nativos. Nenhuma dependência nova de componente externo.

**Stack:** React 18 + TypeScript strict + TailwindCSS v3 + Vitest + cva (mesmo padrão de Story 0.2)

### 3. `FormField` — campo de formulário composto

**Props:**
```ts
interface FormFieldProps {
  label: string
  htmlFor: string
  required?: boolean
  optional?: boolean
  hint?: string
  error?: string
  children: React.ReactNode  // o Input ou qualquer controle
  className?: string
}
```

**Comportamento:**
- Renderiza `<Label>` (átomo) com variante `required` / `optional` / `error` / `default` conforme props
- Renderiza `children` (o Input é passado pelo consumidor — FormField não acoplado ao Input)
- Se `hint` presente: renderiza texto `text-xs text-neutral-400 mt-1`
- Se `error` presente: renderiza texto `text-xs text-accent-600 mt-1` + `<Label>` usa variante `error`
- `error` sobrescreve `hint` visualmente — ambos podem existir nas props mas `error` tem prioridade de exibição
- Wrapper externo: `<div className="flex flex-col gap-1">`

**Exemplo de uso:**
```tsx
<FormField label="E-mail" htmlFor="email" required error="E-mail inválido">
  <Input id="email" state="error" />
</FormField>
```

### 4. `Card` — container de conteúdo

**Props:**
```ts
interface CardProps {
  variant?: 'default' | 'flat' | 'outlined'
  padding?: 'none' | 'sm' | 'md' | 'lg'
  header?: React.ReactNode
  footer?: React.ReactNode
  children: React.ReactNode
  className?: string
}
```

**Variantes de estilo (`variant`):**
- `default` (padrão): `bg-white rounded-lg shadow-elevation-sm` (token do preset)
- `flat`: `bg-white rounded-lg`
- `outlined`: `bg-white rounded-lg border border-neutral-200`

**Padding (`padding`):**
- `none`: sem padding
- `sm`: `p-4`
- `md` (padrão): `p-6`
- `lg`: `p-8`

**Slots:**
- `header`: renderizado acima do body com `border-b border-neutral-100 pb-4 mb-4` (quando presente)
- `footer`: renderizado abaixo do body com `border-t border-neutral-100 pt-4 mt-4` (quando presente)
- `children`: corpo principal do card

### 5. `Alert` — feedback inline persistente

**Props:**
```ts
interface AlertProps {
  variant?: 'info' | 'success' | 'warning' | 'error'
  title?: string
  description?: string
  onDismiss?: () => void
  className?: string
}
```

**Ícones padrão por variante (Lucide via `<Icon>` átomo):**
- `info`: `Info`
- `success`: `CheckCircle2`
- `warning`: `AlertTriangle`
- `error`: `XCircle`

**Tokens de cor por variante (do preset — zero hardcoded):**

| Variante | Background | Border | Ícone / Texto |
|----------|-----------|--------|---------------|
| `info`    | `bg-info-50` | `border-info-500` | `text-info-500` |
| `success` | `bg-success-50` | `border-success-500` | `text-success-600` |
| `warning` | `bg-warning-50` | `border-warning-500` | `text-warning-600` |
| `error`   | `bg-error-50` | `border-error-500` | `text-error-600` |

**Estrutura visual:**
```
[ícone] [título]                     [×]
        [descrição opcional]
```
- Borda esquerda colorida: `border-l-4` na cor da variante
- Dismiss: botão `×` visível só quando `onDismiss` presente, usa `<Icon name="X" />`

**Acessibilidade:**
- `role="alert"` sempre presente
- `aria-live="assertive"` para `error` e `warning`
- `aria-live="polite"` para `info` e `success`

### 6. `Toast` — notificação temporária

**Props:**
```ts
interface ToastProps {
  variant?: 'info' | 'success' | 'warning' | 'error'
  title: string
  description?: string
  duration?: number   // ms, padrão 4000, 0 = sem auto-dismiss
  onClose?: () => void
  className?: string
}
```

**Comportamento:**
- Auto-dismiss: `useEffect` com `setTimeout(onClose, duration)` — executa só se `duration > 0` e `onClose` definido
- Cleanup: cancela `setTimeout` no `unmount` do componente
- Posicionamento: o Toast não define sua própria posição (`fixed`/`absolute`) — o consumidor posiciona
- Visual: mesmo sistema de cor do Alert (mesmos tokens por variante)
- Diferença do Alert: Toast tem sombra `shadow-elevation-sm` e `rounded-lg` mais pronunciado para flutuar sobre o conteúdo
- Botão de fechar (`×`) sempre visível quando `onClose` presente

**Acessibilidade:**
- `role="status"` (não `alert`) — Toast é informativo, não crítico
- `aria-live="polite"`

## Notas técnicas

### Padrão de composição

```tsx
// FormField NÃO cria o Input — o consumidor passa como children:
<FormField label="Senha" htmlFor="pwd" required error={errors.pwd?.message}>
  <Input id="pwd" type="password" state={errors.pwd ? 'error' : 'default'} />
</FormField>
```

Isso mantém FormField agnóstico ao tipo de controle (funciona com Input, Select, Textarea futuros).

Sem FormField, todo módulo implementa seu próprio padrão de validação visual — risco de inconsistência multiplicado por 5 módulos e N formulários. Card e Alert são usados em 90%+ das telas de qualquer SaaS. Toast é o feedback de ação mais comum (save, delete, error de API).

### Observações LOW (não bloqueantes — candidatas a Story 0.3b)

- **[LOW-1]** `role="alert"` em info/success é semanticamente impreciso — WAI-ARIA implica `assertive` independente do `aria-live="polite"` explícito. Solução: `role="status"` para info/success. Implementado conforme spec; imprecisão está na spec, não na implementação.
- **[LOW-2]** `shadow-elevation-sm` ausente no preset — Alert/Toast usam `shadow-sm` do Tailwind padrão. Adicionar token ao `tailwind-preset.js` na Story 0.3b.
- **[LOW-3]** `transition-colors` + `transition-transform` no Button (carryover Story 0.2) — não afeta Story 0.3.
