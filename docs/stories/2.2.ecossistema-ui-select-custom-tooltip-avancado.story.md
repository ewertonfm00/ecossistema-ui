## Story

**Como** desenvolvedor de um módulo consumidor do ecossistema,
**quero** um `SelectCustom` com busca e multi-seleção e um `Tooltip` atualizado com suporte completo a teclado,
**para que** eu possa implementar filtros complexos (busca em lista longa, seleção múltipla com chips) e garantir que tooltips sejam acessíveis para usuários que navegam por teclado.

---

## Contexto

O `Select` nativo (Story 0.2) usa `<select>` HTML — funcional mas limitado: sem busca, sem multi-seleção, sem grupos, sem aparência customizável. Formulários do Estética IA (seleção de procedimentos, especialidades) e ML Lab (seleção de features) precisam de um select com busca e multi-seleção.

O `Tooltip` atual (Story 0.2) só abre em `hover`. Usuários de teclado que nunca dão hover num elemento nunca veem o tooltip — violação WCAG 2.1 SC 1.4.13 (Content on Hover or Focus). A atualização adiciona `trigger="focus"` e `trigger="both"` para suporte completo.

**Decisão arquitetural:** `SelectCustom` implementa o padrão ARIA `combobox` (não `listbox` standalone) pois inclui campo de busca. Sem Radix, sem Select de terceiros — implementação própria seguindo o mesmo padrão das Stories 2.0 e 2.1. O `Select` nativo existente NÃO é removido nem modificado — `SelectCustom` é um componente adicional.

### 2. `SelectCustom` — dropdown customizado com busca e multi-seleção

#### 2a. Interface

```ts
interface SelectOption {
  value: string
  label: string
  disabled?: boolean
  group?: string   // nome do grupo (para agrupamento de opções)
}

interface SelectCustomProps {
  options: SelectOption[]
  value?: string | string[]           // string = single; string[] = multi
  onChange?: (value: string | string[]) => void
  placeholder?: string                // padrão: 'Selecionar...'
  searchable?: boolean                // padrão: false — exibe input de busca
  multiple?: boolean                  // padrão: false — multi-seleção
  disabled?: boolean                  // padrão: false
  clearable?: boolean                 // padrão: false — exibe botão "limpar"
  id?: string
  'aria-label'?: string
  'aria-describedby'?: string
  noOptionsText?: string              // padrão: 'Nenhuma opção encontrada'
  maxDisplayed?: number               // máx de chips visíveis no trigger (multi); padrão: 3
}
```

#### 2b. Comportamento — modo single

**Trigger (campo de exibição):**
- Exibe o `label` da opção selecionada ou o `placeholder` se vazio
- Ícone `ChevronDown` à direita; rotaciona 180° quando aberto
- Se `clearable=true` e há seleção: exibe ícone `X` para limpar (antes do ChevronDown)
- Clique abre/fecha o dropdown
- `aria-haspopup="listbox"` + `aria-expanded={open}` + `aria-controls="listbox-{id}"`

**Dropdown:**
- `position: absolute` relativo ao container (`position: relative`)
- `z-index: 50`
- Se `searchable=true`: input de busca no topo do dropdown com `placeholder="Buscar..."` e ícone `Search`
- Lista de opções filtradas (case-insensitive) pelo termo de busca
- Opção selecionada tem `aria-selected="true"` + ícone `Check` à direita
- Opção `disabled` tem `aria-disabled="true"` + cursor `not-allowed`
- Sem opções (após filtro): exibe `noOptionsText`

**Agrupamento:** se opções têm `group`, separar em grupos com cabeçalho:
```tsx
<div role="group" aria-label={groupName}>
  <div className="px-3 py-1 text-xs font-semibold text-neutral-400 uppercase tracking-wider">
    {groupName}
  </div>
  {groupOptions.map(opt => <OptionItem ... />)}
</div>
```

**Fechar dropdown:** clique fora (blur do container), Escape, seleção de opção.

#### 2c. Comportamento — modo multi (`multiple=true`)

**Trigger:**
- Exibe chips (Badge variant="secondary") para cada valor selecionado
- Se selecionados > `maxDisplayed`: exibe `{maxDisplayed} primeiros chips + "+N mais"` (ex: "+2 mais")
- Se nenhum selecionado: exibe `placeholder`
- Botão `X` dentro de cada chip remove aquela seleção
- `onChange` recebe `string[]` com todos os valores selecionados

**Dropdown:**
- Opções selecionadas têm `aria-selected="true"` + `Check` à direita
- Clicar em opção já selecionada DESSELECIONA (toggle)
- Dropdown NÃO fecha após seleção em modo multi — fecha apenas com Escape ou clique fora
- Checkbox visual opcional à esquerda de cada opção (sem `<input type="checkbox">` — apenas visual com ícone `Square`/`CheckSquare`)

#### 2d. Navegação por teclado (obrigatório — ARIA combobox pattern)

| Tecla | Ação |
|-------|------|
| `Enter` / `Space` (no trigger) | Abre dropdown |
| `ArrowDown` | Move foco para próxima opção (abre dropdown se fechado) |
| `ArrowUp` | Move foco para opção anterior |
| `Home` | Primeira opção |
| `End` | Última opção |
| `Enter` (opção focada) | Seleciona opção |
| `Escape` | Fecha dropdown sem selecionar |
| `Tab` | Fecha dropdown e move foco para próximo elemento |

Quando `searchable=true`, o foco vai automaticamente para o input de busca ao abrir.

#### 2e. Estrutura ARIA (combobox pattern)

```tsx
<div ref={containerRef} onBlur={handleBlur} className="relative">
  {/* Trigger */}
  <button
    role="combobox"
    aria-haspopup="listbox"
    aria-expanded={open}
    aria-controls={`listbox-${id}`}
    aria-labelledby={ariaLabelledby}
    aria-activedescendant={focusedOptionId}
    onClick={toggleOpen}
    onKeyDown={handleTriggerKeyDown}
    disabled={disabled}
    className="..."
  >
    {/* conteúdo do trigger */}
  </button>

  {/* Dropdown */}
  {open && (
    <div id={`listbox-${id}`} role="listbox" aria-multiselectable={multiple} aria-label={ariaLabel} className="...">
      {searchable && <input type="text" role="searchbox" aria-label="Buscar opções" ... />}
      {filteredOptions.map(opt => (
        <div
          id={`option-${id}-${opt.value}`}
          key={opt.value}
          role="option"
          aria-selected={isSelected(opt.value)}
          aria-disabled={opt.disabled}
          onClick={() => handleSelect(opt)}
          onKeyDown={handleOptionKeyDown}
          className="..."
        >
          {opt.label}
          {isSelected(opt.value) && <Icon name="Check" size={16} />}
        </div>
      ))}
    </div>
  )}
</div>
```

### 3. `Tooltip` — atualização com suporte a teclado

**Arquivo a modificar:** `src/atoms/Tooltip/Tooltip.tsx` (existente)

**Adição à interface existente:**
```ts
// Adicionar às props existentes do Tooltip:
interface TooltipProps {
  // ... props existentes (children, content, placement, etc.)
  trigger?: 'hover' | 'focus' | 'both'  // padrão: 'hover' (mantém comportamento atual)
}
```

**Comportamento por `trigger`:**
| Valor | Abre quando | Fecha quando |
|-------|-------------|--------------|
| `hover` (padrão) | `onMouseEnter` no trigger | `onMouseLeave` do trigger |
| `focus` | `onFocus` no trigger | `onBlur` do trigger |
| `both` | `onMouseEnter` OU `onFocus` | `onMouseLeave` E `onBlur` |

**Implementação de `both` (lógica de OR para abrir, AND para fechar):**
```ts
const [isHovered, setIsHovered] = useState(false)
const [isFocused, setIsFocused] = useState(false)
const open = trigger === 'hover' ? isHovered
           : trigger === 'focus' ? isFocused
           : isHovered || isFocused  // 'both'
```

**ARIA (atualização obrigatória):**
O elemento trigger deve ter `aria-describedby` apontando para o id do tooltip:
```tsx
React.cloneElement(children, {
  'aria-describedby': open ? `tooltip-${id}` : undefined,
  onMouseEnter: ...,
  onMouseLeave: ...,
  onFocus: ...,
  onBlur: ...,
})

// Tooltip content:
<div id={`tooltip-${id}`} role="tooltip" ...>
  {content}
</div>
```

**Backward compatibility:** `trigger` não especificado deve manter comportamento exato do Tooltip atual — zero breaking change.

**Stories (Storybook) — adicionar às stories existentes:**
- `FocusTrigger` — `trigger="focus"` — tooltip abre ao dar Tab no botão
- `BothTrigger` — `trigger="both"` — abre em hover E em foco

**Testes (mínimo 2 novos):**
- [ ] `trigger="focus"`: tooltip NÃO aparece em hover, aparece em focus
- [ ] `trigger="both"`: tooltip aparece tanto em hover quanto em focus
- [ ] `trigger="hover"` (padrão): comportamento inalterado (teste de regressão)
- [ ] `aria-describedby` presente no trigger quando tooltip está aberto

## Notas técnicas

### SelectCustom — não reimplementar o Select nativo

O `Select` nativo (`<select>`) continua existindo e sendo exportado. `SelectCustom` é para casos que exigem busca, multi-seleção ou agrupamento. `@dev` não deve modificar o `Select` existente.

### SelectCustom — `aria-activedescendant` para navegação por teclado

Em vez de mover foco real para os `role="option"` (o que quebraria o campo de busca), usar `aria-activedescendant` no combobox trigger apontando para o id da opção "visualmente focada". Isso é o padrão correto ARIA para combobox.

```ts
// Estado interno:
const [focusedIndex, setFocusedIndex] = useState(-1)
const focusedOptionId = focusedIndex >= 0 ? `option-${id}-${filteredOptions[focusedIndex].value}` : undefined

// No button trigger:
aria-activedescendant={focusedOptionId}
```

### SelectCustom — fechar com blur do container

Idêntico ao DatePicker (Story 2.1):
```ts
function handleContainerBlur(e: React.FocusEvent) {
  if (!e.currentTarget.contains(e.relatedTarget as Node)) {
    setOpen(false)
    setFocusedIndex(-1)
  }
}
```

### Tooltip — não usar `cloneElement` se children não for elemento React

Verificar se `children` é um elemento React válido antes de `cloneElement`. Se for string ou fragment, envolver em `<span>` primeiro.

### Tooltip — `aria-describedby` só quando tooltip visível

Aplicar `aria-describedby` apenas quando o tooltip está aberto (`open=true`). Quando fechado, remover o atributo — screen readers não devem anunciar conteúdo invisível.
