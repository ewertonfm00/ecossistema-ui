## Story

**Como** desenvolvedor de um módulo consumidor do ecossistema (Estética IA, Auth, Gestão, ML Lab),
**quero** importar `@aiox/ui` como pacote compilado com CJS + ESM + typings declarativos,
**para que** eu não precise configurar TypeScript para processar `src/` do pacote e o bundle do meu módulo seja tree-shakeable.

## Contexto

O `@aiox/ui` atualmente aponta `"main": "src/index.ts"` e `"types": "src/index.ts"` — sem dist compilada. Consumers precisam incluir o path de `src/` no `tsconfig.json` deles, o que cria acoplamento estrutural e impede tree-shaking via bundlers que não processam TypeScript diretamente (Vite, webpack sem ts-loader).

Esta story configura o pipeline de build com `tsup` e os workflows de CI/Release no GitHub Actions, produzindo artefatos ESM + CJS + `.d.ts` em `dist/`.

**Stack:** `tsup` (zero-config bundler baseado em esbuild) — consistente com o resto do ecossistema AIOX.

### 2. `tsup.config.ts`

```ts
import { defineConfig } from 'tsup'

export default defineConfig({
  entry: ['src/index.ts'],
  format: ['esm', 'cjs'],
  dts: true,
  splitting: true,
  sourcemap: true,
  clean: true,
  external: ['react', 'react-dom'],
})
```

**Notas:**
- `splitting: true` — habilita code splitting por chunk, viabilizando tree-shaking correto no ESM
- `external: ['react', 'react-dom']` — não bundla React (peer dependency)
- `clean: true` — limpa `dist/` antes de cada build
- `dts: true` — gera `dist/index.d.ts` e `dist/index.d.mts`

### 3. `package.json` — campos atualizados

```json
{
  "main": "dist/index.js",
  "module": "dist/index.mjs",
  "types": "dist/index.d.ts",
  "exports": {
    ".": {
      "import": "./dist/index.mjs",
      "require": "./dist/index.js",
      "types": "./dist/index.d.ts"
    },
    "./tailwind-preset": "./tailwind-preset.js"
  },
  "files": ["dist", "tailwind-preset.js", "src"],
  "scripts": {
    "build": "tsup",
    "dev": "tsup --watch",
    "prepublishOnly": "npm run build && npm test -- --run"
  }
}
```

**Notas:**
- `"src"` mantido em `files` para devs que preferem consumir source diretamente
- `exports` com mapa explícito para resolução correta em Node 16+ e bundlers modernos
- `prepublishOnly` garante build + testes passando antes de qualquer publicação npm

### 4. `.gitignore` — adição obrigatória

Adicionar linha `dist/` ao `.gitignore` existente. A dist compilada não entra no git — é gerada no CI.

### 5. `.github/workflows/ci.yml`

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run tests
        run: npm test -- --run

      - name: Build
        run: npm run build

      - name: Verify dist artifacts
        run: |
          test -f dist/index.js || (echo "dist/index.js not found" && exit 1)
          test -f dist/index.mjs || (echo "dist/index.mjs not found" && exit 1)
          test -f dist/index.d.ts || (echo "dist/index.d.ts not found" && exit 1)
```

### 6. `.github/workflows/release.yml`

```yaml
name: Release

on:
  push:
    tags:
      - 'v*.*.*'

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
          registry-url: 'https://registry.npmjs.org'

      - name: Install dependencies
        run: npm ci

      - name: Run tests
        run: npm test -- --run

      - name: Build
        run: npm run build

      - name: Publish to npm
        run: npm publish --access public
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
```

**Nota:** `NPM_TOKEN` deve ser configurado como secret no repositório GitHub. Ewerton configura manualmente no GitHub Settings → Secrets.

## Notas técnicas

### Por que tsup e não tsc diretamente?

`tsc` gera apenas CJS. `tsup` usa esbuild internamente, gera ESM + CJS em paralelo, com `dts: true` delegando para `tsc` apenas os typings. Resultado: build 10-20x mais rápido que `tsc` puro + ESM nativo.

### Verificação manual pós-build

```bash
npm run build
node -e "const { Button } = require('./dist/index.js'); console.log(typeof Button)"
# Output esperado: function
```

### `exports` map e compatibilidade

O campo `exports` tem prioridade sobre `main`/`module` em Node 16+ e bundlers modernos (Vite, webpack 5, esbuild). O fallback `main`/`module` garante compatibilidade com tooling antigo.

## Valor de negócio

Sem dist compilada, cada módulo consumidor precisa configurar TypeScript para processar `src/` de `@aiox/ui` — acoplamento estrutural que aumenta complexidade de cada módulo e impede tree-shaking. Esta story elimina o acoplamento e habilita uso do pacote em qualquer módulo do ecossistema sem configuração extra.
